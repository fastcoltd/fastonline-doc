# HTML Selector Simplification

Use this skill when the task is to simplify verbose HTML selector/class usage in this project, such as replacing repeated inner-node classes with parent-scoped selectors, semantic tags, or stable attributes.

This is a zero-regression refactor. The output must preserve PC and mobile behavior, UI, visual appearance, and layout.

## Core Rules

- Always work from source files first:
  - HTML source: `src/pages` and `src/partials`
  - Shared components: prefer `src/partials/components`
  - Root `*.html` files are generated by `node scripts/build-pages.js`
- Keep `.css` and `.less` synchronized whenever CSS changes.
- Do not change style values, spacing, colors, layout, text, hrefs, images, or DOM hierarchy unless the task explicitly requires it.
- Do not use broad global selectors such as `li {}`, `h2 {}`, `.title {}` without a strong parent scope.
- Do not remove state classes such as `.active`, `.selected`, `.disabled`, `.is-open`, or behavior hooks unless the JS is updated and regression-tested.
- Do not fix unrelated pre-existing behavior bugs during this refactor. Record them separately if discovered.

## Classification

Before editing, classify each class or selector:

- **Parent scope**: stable outer container that safely limits all descendant selectors, e.g. `.header-search-box-label`.
- **Behavior hook**: class used by JS for lookup, event binding, state toggling, or data extraction. Preserve or update JS with equivalent scoped lookup.
- **Reusable style hook**: class shared by multiple modules or variants. Preserve unless all consumers are included in the refactor.
- **Pure structural class**: class only used to reach local child nodes inside a clear parent scope. Candidate for removal.
- **State class**: UI state marker such as `.selected` or `.active`. Preserve behavior and styling.

## Refactor Pattern

Prefer this shape:

```html
<div class="component-root">
    <div class="component-menu">
        <ul>
            <li data-value="items">
                <h2>items</h2>
            </li>
        </ul>
    </div>
</div>
```

```css
.component-root .component-menu [data-value] {
    /* same values as the previous item class */
}

.component-root .component-menu h2 {
    /* same values as the previous title class */
}
```

```js
const menuItems = menuContainer.querySelectorAll('[data-value]');
const title = item.querySelector('.title') || item.querySelector('h2') || item;
```

Use this only when the parent scope uniquely owns the internal structure.

## Workflow

1. Search for candidate selectors with `rg`.
2. Read the relevant HTML, CSS/LESS, and JS together before editing.
3. Identify the smallest safe module boundary.
4. Remove only pure structural classes inside that boundary.
5. Add parent-scoped CSS/LESS selectors with identical declarations.
6. Update JS to use scoped lookups if a removed class was used as a behavior hook.
7. Build root HTML with `node scripts/build-pages.js`.
8. Validate syntax and diffs.
9. Run regression checks from `references/regression-checklist.md`.

## Header Search Example

Before:

```html
<div class="header-search-box-label">
    <div class="header-menu-container">
        <div class="header-menu-selector">
            <ul class="header-menu-list">
                <li class="header-menu-item" data-value="items">
                    <h2 class="title">items</h2>
                </li>
            </ul>
        </div>
    </div>
</div>
```

After:

```html
<div class="header-search-box-label">
    <div class="header-menu-container">
        <div>
            <ul>
                <li data-value="items">
                    <h2>items</h2>
                </li>
            </ul>
        </div>
    </div>
</div>
```

Equivalent CSS must be parent-scoped:

```css
.header-search-box-label .header-menu-container [data-value] {}
.header-search-box-label .header-menu-container h2 {}
```

Equivalent JS should be scoped to the current menu container:

```js
this.menuItems = this.menuContainer ? this.menuContainer.querySelectorAll('[data-value]') : [];
```

## Completion Criteria

- Source and generated root pages are consistent.
- `.css` and `.less` remain synchronized.
- PC and mobile screenshots for the changed area match the baseline.
- Changed interactions still work.
- Nearby shared components or menus that reuse the updated JS still work.
- Any pre-existing behavior difference found during testing is reported as pre-existing, not silently fixed.
